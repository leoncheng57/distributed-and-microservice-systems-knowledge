1. Why Consensus Exists — The Core Problem

The one sentence that matters: Consensus is how a group of unreliable machines, communicating over an unreliable network, can agree on a single value (or an ordered sequence of values) even when some of them crash — without ever producing two conflicting answers.

Every stateful distributed system eventually faces the same question: "who is in charge, and what is the current truth?" If you replicate data across 3 or 5 nodes for durability, you need them to agree on the order of writes. If you elect a leader, you need everyone to agree on which node is leader — and crucially, to never have two leaders accepting writes at the same time ("split brain").

Consensus algorithms solve this by guaranteeing two properties:

  • Safety ("nothing bad happens"): All non-faulty nodes agree on the same value, and that value was actually proposed by some node. Never two conflicting decisions.
  • Liveness ("something good eventually happens"): As long as a majority of nodes are up and can communicate, the system will eventually make progress and decide a value.
Senior SWE framing: You almost never implement consensus yourself in an interview — you delegate it to a battle-tested system (etcd, ZooKeeper, Consul) and reason about its guarantees. The interview win is knowing what it gives you (linearizable writes, leader election, distributed locks), what it costs (latency, majority availability), and how it fails (network partitions, leader churn).

2. Fundamental Mental Models

Leader-based consensus

Raft / Multi-Paxos / ZAB

One node is elected leader and serializes all writes. The leader appends entries to a replicated log and waits for a majority to acknowledge before committing. This means:

  • Simple ordering: The leader assigns a total order to all operations
  • Fast steady state: One network round-trip to a majority per write
  • Election on failure: If the leader dies, a new one is elected (brief unavailability)
  • This is what powers etcd, Consul, ZooKeeper, CockroachDB, TiKV
Leaderless / quorum consensus

Classic (Single-Decree) Paxos

Any node can propose; agreement is reached through two-phase voting (prepare/promise, then accept/accepted) using majority quorums. This means:

  • No single point of failure for proposing, but more message rounds
  • Dueling proposers can stall progress (livelock) without a leader
  • Theoretical foundation — most real systems use a leader-based variant (Multi-Paxos)
  • Famously hard to understand — which is exactly why Raft was created
Replicated State Machine (RSM) — the unifying idea: If every node starts in the same state and applies the same commands in the same order, they all end in the same state. Consensus is "just" the mechanism to agree on that ordered log of commands. Raft, Paxos, and ZAB are all different ways to build the same replicated log.
Fig 1. The replicated state machine model. Consensus produces an agreed-upon ordered log; each node applies it deterministically to reach identical state.

3. Quorums & Why Majorities Matter

The beating heart of every consensus protocol is the majority quorum: a decision requires acknowledgment from more than half the nodes — ⌊N/2⌋ + 1. The magic property: any two majorities always overlap in at least one node. That overlapping node "remembers" the previous decision, which is what prevents two conflicting decisions.

Fig 2. Majority quorums always intersect — the foundation of consensus safety and split-brain prevention.

Cluster Sizing — Why 3 and 5, Never 4

Senior SWE pitfall — always use odd numbers: A 4-node cluster tolerates the same single failure as a 3-node cluster but requires a larger quorum (3 vs 2) for every write, so it's strictly worse. Adding nodes increases fault tolerance but increases write latency because the leader must wait for more acknowledgments. Bigger is not faster.

4. Raft — Leader Election & Log Replication

Raft was designed in 2014 explicitly for understandability. It decomposes consensus into three sub-problems: leader election, log replication, and safety. Every node is always in one of three states.

Fig 3. Raft's three states. Followers become candidates on election timeout; candidates become leaders by winning a majority of votes.

Terms — Raft's Logical Clock

Raft divides time into terms, numbered consecutively. Each term begins with an election. A term has at most one leader. Every RPC carries the sender's term; if a node sees a higher term, it immediately reverts to follower. This is how Raft detects stale leaders and prevents split brain — an old leader that was partitioned away will have a lower term and be rejected the moment it reconnects.

Leader Election Mechanics

  • Followers expect periodic heartbeats (empty AppendEntries) from the leader.
  • If a follower hears nothing for its randomized election timeout (typically 150–300 ms), it becomes a candidate, increments the term, votes for itself, and sends RequestVote RPCs.
  • A node grants its vote (one per term) if the candidate's log is at least as up-to-date as its own.
  • Winning a majority → leader. The randomized timeouts make simultaneous candidacies (split votes) rare, so elections usually resolve in one round.

Log Replication — The Steady State

Fig 4. Raft log replication. An entry is committed once a majority of nodes have persisted it; only then is it applied and acknowledged to the client.
Reads are subtle too: A naive leader could serve a stale read if it was just deposed by a partition it hasn't noticed. Safe linearizable reads require the leader to confirm it's still leader — via a ReadIndex (confirm commit + heartbeat a majority) or leader leases (time-bounded leadership). Many systems offer a faster "stale read from follower" mode when you don't need linearizability.

5. Raft Safety: Terms, Commit Rules & Edge Cases

Election Restriction

A candidate can only win if its log is at least as up-to-date as a majority. Up-to-date is defined by (last log term, last log index). This guarantees a new leader already contains all committed entries — committed data is never lost in an election.

Leader Append-Only

A leader never overwrites or deletes entries in its own log; it only appends. Conflicting follower entries are forced to match the leader's log (followers truncate and copy), never the reverse.

Commit-from-Current-Term Rule

A leader cannot commit an entry from a previous term just because it's replicated on a majority. It must commit a current-term entry (which carries the older ones with it). This closes a famous subtle bug where an old entry could be overwritten after appearing committed.

The classic interview trap — "doesn't a majority ack mean committed?": Not always. If a new leader sees an old-term entry replicated on a majority, it still can't declare it committed, because a future leader without that entry could still win an election and overwrite it. Raft only considers an entry committed once an entry from the leader's current term is committed. Knowing this nuance signals deep understanding.

Log Matching Property

Raft maintains a powerful invariant: if two logs contain an entry with the same index and term, then the logs are identical in all preceding entries. AppendEntries includes the index/term of the entry immediately preceding the new ones; a follower rejects the request if it doesn't match, and the leader walks backward until they reconcile. This consistency check is what keeps all logs convergent.

Cluster Membership Changes

You can't just swap the config on all nodes at once — during the transition you could momentarily form two disjoint majorities (split brain). Raft solves this with joint consensus (decisions require majorities in both old and new configs during the overlap) or, more commonly in practice, single-server changes (add/remove one node at a time so quorums always overlap).

Log Compaction / Snapshots

The log can't grow forever. Nodes periodically take a snapshot of the state machine and discard log entries up to that point. A leader sends an InstallSnapshot RPC to a follower that has fallen so far behind that the needed entries were already compacted away.

6. Paxos & Multi-Paxos

Paxos (Lamport, 1998) is the original proven-correct consensus algorithm. Single-decree Paxos agrees on one value through three roles — proposers, acceptors, and learners — using two phases.

Fig 5. Single-decree Paxos. The "re-propose an already-accepted value" rule is what guarantees safety across competing proposers.

Multi-Paxos — Making It Practical

Running full two-phase Paxos for every single value is expensive. Multi-Paxos elects a stable leader that skips Phase 1 for subsequent values (it already holds the promise), reducing steady-state cost to a single round-trip — effectively the same as Raft. In fact, Raft can be seen as a more prescriptive, understandable Multi-Paxos.

Paxos strengths
  • Formally proven, decades of theory
  • Maximally flexible / general
  • Powers Google Chubby, Spanner
Paxos weaknesses
  • Notoriously hard to understand & implement correctly
  • The paper leaves many practical gaps (the "Multi-Paxos" details)
  • Dueling proposers can livelock without a leader
Raft vs Paxos — the honest take: They provide the same guarantees and the same steady-state performance. Paxos came first and is the theoretical bedrock; Raft reorganized the same ideas around a strong leader and an understandable spec, which is why almost every new system since 2014 (etcd, Consul, CockroachDB, TiKV, Kafka KRaft) chose Raft. If asked "which would you use," the answer is almost always "a mature Raft implementation."

7. ZAB, Viewstamped Replication & Byzantine Fault Tolerance

Leader-based

ZAB (ZooKeeper Atomic Broadcast)

The protocol behind Apache ZooKeeper. Like Raft: a single leader, totally-ordered broadcast of state changes, majority quorums. Optimized for high read throughput and a primary-backup model with epoch numbers (analogous to Raft terms).

+ Strengths
  • Battle-tested (Hadoop, Kafka pre-KRaft)
  • Fast recovery
  • Strong ordering guarantees
- Weaknesses
  • Tied to ZooKeeper
  • ZK ops can be heavy
Leader-based

Viewstamped Replication (VR)

Predates Paxos's popularity; rediscovered as essentially equivalent. Uses views (like terms/epochs) and a primary. Historically important — shows leader-based consensus was independently invented multiple times.

+ Strengths
  • Clean primary/backup model
  • Influenced modern designs
- Weaknesses
  • Less industry adoption by name
Byzantine fault tolerant

PBFT / BFT-family

Raft and Paxos tolerate crash faults (nodes stop) but assume non-faulty nodes are honest. Byzantine protocols tolerate nodes that lie, send conflicting messages, or are compromised. Needs 3f+1 nodes to tolerate f malicious ones.

+ Strengths
  • Tolerates malicious/arbitrary faults
  • Foundation of blockchains
- Weaknesses
  • More nodes & messages (O(n²))
  • Overkill inside a trusted datacenter
Crash-fault vs Byzantine — know the distinction: Inside your own datacenter, nodes are trusted; you only worry about crashes and network issues → use Raft/Paxos (tolerate f failures with 2f+1 nodes). Across untrusted parties (cryptocurrencies, consortium ledgers), nodes may be adversarial → you need BFT (tolerate f with 3f+1 nodes). Don't bring BFT to a datacenter problem.

8. FLP Impossibility & CAP Connection

FLP Impossibility (1985)

In a fully asynchronous network (no bound on message delay) with even one crash-faulty process, no deterministic consensus algorithm can guarantee it always terminates.

So how do Raft/Paxos work? They sacrifice guaranteed liveness, not safety. They use timeouts (a partial-synchrony assumption) to make progress in practice. Under pathological timing they could theoretically loop forever (e.g., endless split votes), but randomization makes this vanishingly unlikely.

CAP Theorem Connection

Consensus systems are CP: under a network Partition, they choose Consistency over Availability. The minority side of a partition cannot form a quorum, so it stops accepting writes (and optionally reads) rather than risk diverging.

This is why a 3-node etcd cluster split 2–1 keeps serving on the majority side and rejects writes on the lone node. Availability is sacrificed to preserve a single source of truth.

The senior-engineer take: "Safety is never violated; liveness is best-effort." A consensus system will never give you two conflicting committed values, but it can become temporarily unavailable (no leader, no quorum). When you choose Raft/etcd, you are explicitly choosing "rather be down than wrong." For data where availability matters more than strict consistency (shopping carts, social feeds), you'd reach for an AP, eventually-consistent store instead (Dynamo-style).

9. Where Consensus Is Used in Real Systems

Coordination plane

Configuration, locks & leader election

  • etcd (Raft) — the brain of Kubernetes; stores all cluster state
  • ZooKeeper (ZAB) — coordination for Hadoop, HBase, legacy Kafka
  • Consul (Raft) — service discovery, config, distributed locks
  • Distributed locks & leader election for your own services
Data plane

Replicated databases & logs

  • CockroachDB / TiKV / YugabyteDB — Raft per data range/shard
  • Google Spanner — Paxos groups per shard (with TrueTime)
  • Kafka KRaft — Raft replaces ZooKeeper for metadata
  • Replicated write-ahead logs for HA databases
Rule of thumb: Reach for consensus when you need strong, single-source-of-truth agreement — who is leader, what is the committed order of writes, who holds the lock. Keep the consensus group small (3–5 nodes) and put high-volume data outside it, sharded across many independent consensus groups (this is exactly how CockroachDB/Spanner scale).

11. Production War Stories & Gotchas

The Even-Number Trap

Someone "adds a node for more reliability," going from 3 to 4. Now you need 3 acks instead of 2 for every write (slower) while still tolerating only 1 failure. Worse, a 2–2 partition has no majority at all → total outage.

Fix: Always run odd-sized clusters (3, 5, 7).

Disk fsync Lies

Consensus durability assumes a log entry is truly on stable storage before acking. If fsync is batched/lied about (cheap cloud disks, misconfigured DBs), a leader can ack then lose the entry on crash → committed data loss.

Fix: Ensure real fsync; understand your storage's durability semantics.

Election Storms / Flapping Leaders

Election timeout too short relative to network RTT (or GC pauses / CPU starvation) → followers prematurely time out, trigger elections, leadership thrashes, throughput collapses.

Fix: Tune election timeout >> heartbeat interval >> RTT; isolate the consensus process from noisy neighbors; watch GC pauses.

The Stale Read

A partitioned old leader still thinks it's leader and serves a read with stale data before noticing it lost quorum.

Fix: Use ReadIndex or leader leases for linearizable reads; explicitly opt into "stale reads" only when you can tolerate them.

Cross-Region Latency

Putting consensus members across distant regions means every write pays a cross-region round-trip to reach a majority — tens to hundreds of ms per write. People are shocked their "fast" DB is slow.

Fix: Keep a quorum within one region, or accept the latency consciously; consider witness/learner replicas that don't vote.

The Lost Majority / Wedged Cluster

Lose 2 of 3 nodes (or 3 of 5) and the cluster cannot elect a leader or accept writes — it's down by design, not a bug. Recovery from a dead majority is dangerous (risk of data loss).

Fix: Spread nodes across failure domains (AZs); have a tested, documented disaster-recovery runbook.

12. System Design Interview Cheat Sheet

The winning pattern: Don't recite the Raft paper. State the requirement ("we must never have two leaders / we need a consistent committed order"), name the property (majority quorum, leader election, linearizable log), then pick a proven system (etcd/ZooKeeper) and acknowledge the cost (CP — unavailable on minority partition, write latency = round-trip to majority).

Example: "For leader election among my workers I'll use etcd's lease + compare-and-swap rather than building Raft myself. It gives me linearizable leadership with automatic failover; the trade-off is the etcd cluster must keep a majority up, and elections add a few hundred ms of failover time."
